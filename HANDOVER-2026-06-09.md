## WHAT THIS PROJECT IS

A full rebuild of the Peninsula Youth Theatre website. PYT is a non-profit youth theatre in Mountain View, California. The goal is a **free-hosted, mobile-first website that non-technical staff can edit through web forms** — no GitHub, no code, no developer needed for day-to-day content.

**Stack:**
- **Astro 6** — static site generator. Pages in `src/pages/`. Content collections (Shows, Programs, Stories on Stage, Cast Pages) use Astro content collections with Zod schemas in `src/content.config.ts`. Single-record settings (home-page.json, donate-page.json, etc.) are NOT collections — they're plain JSON imported directly into the page that uses them.
- **Decap CMS 3.10.1** — the admin at `/admin`. Config at `public/admin/config.yml`. (Decap script is in `<body>`, not `<head>` — there's a hard-won reason; see PROJECT_RULES.)
- **Cloudflare Pages** — free hosting, auto-deploys from GitHub `main`. OAuth handled by two Cloudflare Pages Functions in `functions/api/`.

**Key URLs:**
- Live test site: `https://pyt-website.pages.dev`
- CMS login: `https://pyt-website.pages.dev/admin`
- Repo: `https://github.com/PYTheatre/pyt-website` (org is **PYTheatre** — NOT the empty `peninsula-youth-theatre` org; don't be confused by it)
- Real domain (future, not live yet): `pytnet.org`

## ⚠️⚠️ UPLOADS — THE SINGLE BIGGEST SOURCE OF PAIN. READ CAREFULLY.

The upload step has repeatedly gone wrong and cost the PM and Claude a lot of time. Internalize these:

**1. For single-file or few-file changes, prefer EDIT-IN-PLACE over uploading.** Give the PM a direct GitHub edit link, e.g.
`https://github.com/PYTheatre/pyt-website/edit/main/src/components/Header.astro`
and tell them: click in the editor, Cmd+A to select all, delete, paste the new contents, Commit. This sidesteps every folder-path trap. **This is the recommended method for editing an existing file.**

**2. For a BRAND-NEW file, use a folder-targeted upload link** so the uploader drops it in the right place:
`https://github.com/PYTheatre/pyt-website/upload/main/src/pages` (then they drop the file there).
**NEVER tell the PM to "drag the folders into GitHub."** The web uploader drops loose files at the repo ROOT, losing folder paths. On 2026-06-08 this caused files to land at the root and in a junk nested folder, breaking the build repeatedly.

**3. The "rename trick" mangles paths.** Typing a path with slashes into the filename box is fragile and concatenates into garbage if the old text isn't cleared. Avoid it. Prefer edit-in-place or folder-targeted upload.

**4. Uploads can ADD or OVERWRITE but CANNOT DELETE.** For any deletion, give a direct delete link:
`https://github.com/PYTheatre/pyt-website/delete/main/path/to/file` → PM commits it.

**5. config.yml is the fussiest file in the project; YAML breaks on bad indentation.** When the PM must paste a block INTO config.yml (not replace it — e.g. adding one CMS record), this is the highest-risk upload there is. Give the exact insertion point by naming the records it goes between, tell them the block's indentation is already correct and to paste as-is without adding/removing leading spaces, and warn that if the build goes red afterward it's almost certainly the paste's indentation. Always verify the block parses as YAML on your end first.

**6. When something "doesn't show up," DIAGNOSE, don't guess. First move: read the Cloudflare build log.** On 2026-06-08 many rounds were wasted re-uploading when one line of the build log showed the real cause (a missing imported file). Pattern:
   - Ask the PM: Cloudflare → the project → Deployments → newest row → **Details** → scroll to the bottom of the build log → paste the last ~15 lines.
   - A successful build that "doesn't show" is almost always (a) the file landed in the wrong place, or (b) browser cache. Have them check the repo folder directly and hard-refresh / use incognito.
   - A red ✗ on a commit = the build failed; the site keeps serving the last good build, so nothing changes no matter what you fix until it goes green.

**7. Decap CMS config caching:** after ANY `public/admin/config.yml` change, new CMS fields won't appear until the PM hard-refreshes `/admin` (Cmd+Shift+R) or uses incognito. Tell them proactively every time.

**8. The "0 files changed" trap:** if the PM drags in stale copies from Downloads, GitHub commits nothing. Give freshly-named/dated files and have them clear old downloads.

**9. Verify success** by the commit's file list ("N files changed", not 0) and by checking the file is in the RIGHT FOLDER on GitHub before assuming it worked.
