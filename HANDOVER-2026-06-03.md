### Upload gotchas (learned the hard way — these recurred THIS session, take them seriously)

- **Use folder-specific upload URLs.** Give me links like `https://github.com/PYTheatre/pyt-website/upload/main/src/pages/shows`, not just the repo root. When I upload a file from the root page, GitHub can strip the folder and dump it at the top level. This bit us repeatedly.
- **Renamed-copy trap.** If I download a file twice, my Mac names it `config (2).yml` or `content.config (3).ts`. Uploaded as-is, it lands with the wrong name and the real file is untouched (so the change silently doesn't take effect). Tell me to use a FRESH download and check the filename has no `(2)`/`(3)` suffix.
- **"0 files changed"** on commit = I dragged a stale/duplicate file. The commit must say "N files changed," not 0.
- **Chat-and-upload can only ADD or OVERWRITE, never DELETE.** Deletions must be done by me directly on GitHub. When something needs deleting, give me a direct `.../delete/main/path` link.
- **Decap caches its config.** After any `config.yml` change, remind me to hard-refresh `/admin` (Cmd+Shift+R) or use incognito, or new CMS fields won't appear.
- **Don't include `src/content/**` files in uploads** unless I explicitly ask to overwrite live content — I maintain that through the CMS and your sandbox copy is stale (PROJECT_RULES Rule 2). (One deliberate exception this session: a `test-cast-page.md` we needed to seed before the CMS could make clean pages — flagged at the time.)
- **Cloudflare "internal error" deploy failures** are usually transient — retry once before debugging.
- **`package.json` + `package-lock.json` must travel together** if a dependency changes, or the Cloudflare build fails for lack of the library.

## TROUBLESHOOTING CLOUDFLARE DEPLOYS

dash.cloudflare.com → Workers & Pages → pyt-website → Deployments. Top row = newest. Green tick = success; yellow = building; red = failed. For a failure: click "Details" → Build log → scroll to the bottom red error ("Copy log"). A failed build keeps the previous good version live, so new changes won't appear until a build passes.

**A recurring symptom worth knowing:** if a URL that has no real page is visited (e.g. `/shop` before we made it, or a cast page at a garbled slug), Cloudflare serves the **home page** as a fallback. So "this page shows the home page content" usually means "that page/route doesn't exist," not "content got duplicated." We hit this twice this session.
