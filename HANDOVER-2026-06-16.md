## 1. What this project is

A complete rebuild of the **Peninsula Youth Theatre (PYT)** website — a Bay Area community youth theatre. The old site was a dated, hard-to-edit WordPress site. The new site is a fast, statically-generated site that PYT staff can edit themselves through a simple admin panel (`/admin`), with no developer involvement.

**The PM's role:** non-technical project manager. They make all content/design decisions, do the actual uploads to GitHub, and do every live visual check (Claude cannot see the live site). They rely on Claude to build, to explain choices in plain terms, and to be honest about what's been tested.

**The end goal:** carry the site to launch on the real domain (`pytnet.org`). Most features are built and live. The big remaining work is the rebrand + go-live (Phase 5), a few items blocked on third-party setup, and a staff "how to edit the site" guide.

---

## 2. The stack, in plain terms

- **Astro 6** — the site generator. Pages: `src/pages/`. Reusable content (shows, programs) lives in "content collections" with a schema at `src/content.config.ts`.
- **Decap CMS 3.10.1** — the admin panel at `/admin`. One config file controls every editable field staff see: `public/admin/config.yml`.
- **Cloudflare Pages** — the host. Every push to `main` on GitHub triggers an auto-rebuild (~2 min). Live URL today: **`pyt-website.pages.dev`**. Real domain `pytnet.org` comes in Phase 5.
- **GitHub repo:** `PYTheatre/pyt-website` (branch `main`). Ignore the empty `peninsula-youth-theatre` org — it's a red herring.
- **Editable content vs structure:** staff edit *words, images, links, numbers* through the CMS. *Layout/structure* stays in code. Locked decision.

**Third-party services:** NonprofitSoapbox (donations → syncs to Salesforce; this is *why* Soapbox is kept), MailChimp (newsletter), Active.com (class registration), SmugMug (photos), Shopify (shop only now), Google Sheets/Forms (cast schedules, audition conflicts).

- Tell the PM to **watch the Cloudflare deploy go green (✓)**. A red ✗ = build failed, live site keeps serving the old version (see §6A).

## 6. CHALLENGES & RECURRING TRAPS (read carefully)

**A. "CMS edits not showing on the live site."** Instinct: blame the CMS↔site link. **Real cause (happened this session):** a code bug made the Cloudflare **build fail**, so the live site kept serving the last good version and silently ignored every later CMS commit. The red ✗ on the GitHub commit list / Cloudflare deploys is the tell. **When the PM says "my changes aren't showing," check Cloudflare deploy status (green ✓ vs red ✗) and the GitHub commit list FIRST**, before theorizing about the CMS. A failed build freezes the whole site.

**B. "Can't clear a field in the CMS" trap (hit twice this session).** Optional text fields with a **default value** in code can't be cleared: emptying the field makes Decap drop the key, and code written `field || "Default"` (or `field !== undefined ? field : "Default"`) falls back to the default — re-showing what they tried to remove. **Fix that works: pure opt-in** — `const x = data.field || ""` and render only `{x && <p>{x}</p>}`. No default. We applied this to the home-page tags and the Support-page eyebrows. **If the PM says "I cleared a field but it won't go away," this is why.**

**C. Stale sandbox content (Rule 2, ongoing).** The PM edits content live in the CMS constantly. The sandbox `src/content/**` files are out of date (e.g. this session the home-page thermometer showed 50% in the sandbox vs 62% live; Support-PYT tier names/amounts differed). **Never overwrite live content files.** When the PM says "the CMS doesn't match the page," the answer is usually: live content is the truth, edit it in the CMS — not in code. If they want code seed-data aligned to live, ask them to paste current live values.

**D. Hardcoded vs CMS-editable confusion.** Recurring: copy that looks editable is hardcoded in a page template (e.g. the old "Giving levels" heading, the Judy Robe sentence, the entire Photos page). Pattern: PM can't find something in the CMS → it was never wired up → make it editable. Default to making things editable when asked.

**E. Decap field-clearing behaviour can't be verified from the sandbox.** Claude can't run the live CMS, so whether Decap saves `""` vs drops a key on clear is the PM's live check. Be honest. The opt-in pattern (B) handles both, so prefer it.

**F. grep/verification false alarms.** Several times a raw `grep -c` count looked alarming ("2 EIN notes!", "0 Join buttons!", "thermometer at 50%!") but was matching the footer, the `<style>` block, HTML-encoded text, or stale sandbox data. **Verify by inspecting the actual rendered context, not raw counts, before reporting a problem.**

**G. YAML whitespace when editing `config.yml`.** Blank lines in `config.yml` carry a single leading space, so `str_replace` blocks that include "blank" lines can fail to match. For big reorders, a small Python script that splits on record markers is more reliable than a giant str_replace (used this for the Support-PYT reorder).

**H. The general "fix from memory" trap (Rule 1)** — still the biggest historical risk. Stay disciplined: diagnose against reality (deploy status, reproduce in sandbox), don't guess.

- **`DECISIONS.md` was updated this session** to record this (the old Shopify model is collapsed under a "SUPERSEDED" note). Trust the updated `DECISIONS.md`.

## 10. RISKS IN THIS CONTEXT TRANSITION (and how to bridge them)

1. **Stale sandbox content is the #1 risk.** A fresh Claude clones the repo and may assume the content files reflect the live site. They often don't. **Bridge:** treat `src/content/**` as reference-only, never include it in uploads, ask the PM to confirm current live values when in doubt. Re-read Rule 2.

2. **The new Claude lacks the "why."** Why sponsorship is Soapbox not Shopify; why eyebrows are opt-in; why the Photos page suddenly got a CMS record; why the CMS labels look the way they do. **Bridge:** §5–§7 capture the reasoning. Read this whole file + `PROJECT_RULES.md` + `DECISIONS.md` before touching anything.

3. **The CMS labels changed but the underlying record/file names did not.** A new Claude seeing "Support PYT — Giving Levels" in the CMS must know that's still `donor-tiers.json` in code. **Bridge:** §5 spells out the label↔file mapping explicitly.

4. **The 20-page build canary.** If a change makes it 19 or errors, something broke. **Bridge:** run `npm run build` at session start (confirm clean baseline) and after every change; treat a page-count change or error as a stop signal.

5. **The PM does all visual verification.** Claude can't see results. **Bridge:** always tell the PM exactly what to look at after each deploy; never claim a visual thing "looks right" — only that the build passed and the HTML contains what's expected.

6. **Failed-build silence.** If "changes aren't showing," check Cloudflare ✓/✗ and GitHub commit ✗ FIRST (§6A).

Net: low-risk *if* the new Claude reads this file + `PROJECT_RULES.md` + `DECISIONS.md` first, runs a baseline build, and respects the stale-content rule. The biggest danger is a confident new Claude moving fast on assumptions. The culture of this project: slow down, verify against reality, plan, ask, test, record.
